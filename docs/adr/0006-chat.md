# 0006 - Chat: conversations, messages, and the Telegram-style web UI

- Status: Accepted
- Date: 2026-07-03

## Context

OpenPlays needs messaging so players can coordinate around games without leaving the app: direct
messages between users, and a group chat per play shared by its roster. Messages should feed the
existing notification system (feed + Web Push, ADR 0004) so recipients hear about them, and chat
should degrade gracefully — the stack is SSR-first SvelteKit over a Go/SQLite API, with no realtime
transport (WebSocket/SSE) available yet.

Several sub-problems needed decisions:

- How conversations come into existence, and who may participate — especially for play chats, where
  the roster changes over time.
- How to paginate two differently-shaped conversation sources (DM and play) behind one list API, and
  how to paginate message history.
- How much rich content to store: users paste URLs, and links shared by strangers are a phishing
  surface.
- How "live" the UI can feel (own sends, other people's messages) without sockets.
- Chat UI layout and scrolling behavior, where browser support for bottom-anchored scrolling is
  inconsistent. The visual and interaction reference is Telegram Web A (telegram-tt).

## Decision

### Data model and access

1. Store conversations in `chat_conversations` with `kind` of `dm` or `play`; DMs carry a unique
   `dm_key` (sorted user-id pair), play chats a unique `play_id`.
2. Create conversations idempotently on first open (`ON CONFLICT` no-op), via
   `POST /api/chat/dms` and `POST /api/chat/play-conversations`.
3. Derive play-chat membership live from the current roster (hosts, creator, `confirmed`/`added`
   participants) instead of a membership table; access follows the roster as it changes.
4. Enforce user blocks for DMs: blocked pairs cannot create or see each other's DM conversations.
5. Store messages as plain text (max 2000 chars) in `chat_messages` with a global autoincrement id;
   delete is a sender-only soft delete (`body` nulled, `deleted_at` set).
6. Track read state as a per-user cursor (`last_read_message_id`) in `chat_read_states`; unread
   count = others' non-deleted messages above the cursor.

### API shape

7. Paginate messages with a keyset cursor: `before_id` over `id DESC`, page size 50, returned
   ascending.
8. Serve one merged conversation list (DMs + play chats) sorted by last activity, paginated with a
   composite `(activity_at, id)` cursor applied identically to both source queries, fetching
   `limit + 1` to emit `next_cursor`.
9. Notify chat messages through the ADR 0004 pipeline as kind `chat.message` (debounced feed +
   push), linking to `/chat/{conversation_id}` for both DMs and play chats; play recipients are the
   current roster minus the sender.

### Content

10. Store no markup: message bodies hold exactly the typed characters. All formatting is derived at
    render time through a generic parser pipeline (body → spans → segments → renderers); linkify is
    the first parser, matching `http(s)://` URLs only.
11. Gate link opening behind a confirmation dialog showing the full URL, then open in a new tab
    with `noopener,noreferrer`.

### Frontend architecture

12. Keep chat SSR-first: server `load` functions fetch data; send/delete are form actions using the
    POST-redirect-GET pattern, progressively enhanced with `use:enhance`.
13. Intercept same-page redirects from chat actions and refresh data in place (`invalidateAll`)
    instead of letting enhance run a `goto()`, which would scroll the window and drop focus.
14. Render sends optimistically: a module-level pending store adds a translucent bubble (negative
    local id) the moment the form submits; the entry is removed after the refreshed data arrives,
    and failures restore the unsent text into the composer.
15. Hide conversations from everyone's list until they contain a message (except the one currently
    open), so opening a chat without sending never clutters other members' lists.
16. Accept action-driven freshness for now: data refreshes on navigation, send/delete, and
    notification clicks (including an explicit in-place refresh when the notification targets the
    conversation already open). No polling or sockets yet.

### Layout and scrolling

17. Lock chat routes to the viewport (`h-dvh`, footer omitted, `main` bounded) so the document
    never scrolls, telegram-tt style; the message list is the only scroll container.
18. Use separated rounded cards: the conversation list is a card; the conversation pane has no
    background — the header bar and composer float over the message list, which runs to the
    viewport edges and fades out under them via gradients.
19. Anchor the message list with `flex-direction: column-reverse` (opens at newest, sticks to
    bottom natively) and correct scroll position in JS when prepending history, by measuring an
    anchor message's on-screen position before and after (telegram-tt's technique).
20. Load older pages via an IntersectionObserver sentinel (200px margin) that keeps fetching while
    it remains in view, since observers only fire on intersection transitions.
21. Carry messages that slide out of the refreshed latest-50 window into the client-side history
    cache, so cached scrollback never develops a gap.

## Rationale

### Why conversations are created idempotently on open

The alternative — creating only when the first message sends — needs draft routes and
lookup-without-create endpoints for each conversation kind. Idempotent creation keeps entry points
trivial (`profile → Message`, `play → Go to chat` are one POST + redirect) at the cost of empty
rows, which are bounded (one per DM pair / play) and hidden from lists until a message exists
(decision 15). The visible behavior matches "chat exists once someone speaks" without the
machinery.

### Why play membership is derived, not stored

A membership table would need syncing on every roster change (join, leave, host add/remove,
waitlist promotion). Deriving membership from the roster at query time means access always matches
the roster right now — someone who leaves a game loses the chat immediately, someone added gains
it — with no sync code to break.

### Why plain text and render-time formatting

Storage should hold what the user authored; links are a presentation inference. Deriving them at
render time upgrades all history whenever the parser improves, keeps every consumer of `body`
(previews, notifications, search) trivially safe, and avoids a stored-markup sanitization contract.
A Telegram-style stored-entities design was prototyped and deliberately reverted: entities only pay
off when users author formatting that plain text cannot carry (custom link labels, bold). The
render pipeline is shaped so that migration stays additive. Media, when it comes, is a separate
message kind/attachment concept — not markup — in line with how Telegram models it.

### Why links confirm before opening

Chat is the first surface where users see URLs authored by other users. The `http(s)`-only parser
prevents scheme-based injection, and the confirmation dialog showing the full URL counters
look-alike and pasted-link phishing; `noopener,noreferrer` prevents the opened page from scripting
against the app window.

### Why optimistic send without a full local message store

Telegram keeps a normalized client-side store (`byId` + id ranges) because updates are pushed and
clients must merge everything. Without a realtime transport, our source of truth refresh is the
invalidated server load; a pending-entry overlay achieves instant perceived sends with a fraction
of the machinery. The two store-like behaviors that matter today are handled narrowly: dedupe by id
when the refreshed window overlaps cached history, and window carry-over (decision 21) so refreshes
cannot create gaps. If polling or sockets arrive, these merge points are where a real store slots
in.

### Why column-reverse plus one JS correction

In a column-reverse scroll container the browser treats the bottom as scroll origin, giving
open-at-newest, stick-to-bottom, and stay-put-while-reading-history without any scripting — the
behaviors telegram-tt implements in JS wholesale. The one case browsers disagree on is prepending
content (history pages), so that single case gets the explicit anchor-measure-and-adjust
correction. Verified against telegram-tt source: they scroll entirely in JS with
`overflow-anchor: none`; our approach is strictly less JS for the same UX.

### Why keyset cursors

Offset pagination breaks when rows are inserted while paging — precisely the chat workload.
`before_id` is a natural message cursor because ids are monotonic. The conversation list sorts by
`COALESCE(last_message.created_at, updated_at)`, which changes as activity happens, so its cursor
is the composite `(activity_at, id)` matching the sort; applying it inside both source queries
keeps the in-Go merge correct.

### Why in-place refresh instead of enhance's default redirect handling

Chat actions redirect back to the conversation (PRG for the no-JS path). Applying that redirect as
a client-side `goto()` scrolls the window to the top and moves focus to `<body>` on every send —
unusable for chat. Detecting the same-page redirect and calling `invalidateAll()` preserves scroll
and keeps focus in the composer, so consecutive sends need no re-clicking.

## Consequences

### Positive

- Chat works end to end without JavaScript (forms + PRG), and enhances to app-like behavior.
- Own actions feel instant (optimistic sends, in-place refresh); no scroll or focus jumps.
- Play chat access is always consistent with the roster, with zero sync code.
- Message and conversation pagination are stable under concurrent inserts.
- Message bodies are safe plain text everywhere; the link surface is scheme-restricted and
  confirm-gated.
- The formatting pipeline and notification integration extend without storage changes.
- Scrolling behavior is mostly native CSS; JS touches only the history-prepend case.

### Trade-offs

- No passive freshness: another user's message appears only after a notification click, an own
  action, or navigation. Polling (reusing the in-place refresh path) is the planned next step;
  sockets later.
- The latest-50 window plus cached scrollback is not a real message store; carried-over and cached
  messages are frozen snapshots (a message deleted after leaving the window shows stale until
  reopen).
- Deriving play membership at query time makes the conversation list query heavier than a
  membership-table join; acceptable at current scale.
- Conversations hidden until first message means an opened-but-silent chat also disappears from the
  opener's own list; they re-enter from the game page or profile.
- The IntersectionObserver load loop can overshoot by one page because observer callbacks arrive
  between frames.
- Old notification rows created before the play-chat URL fix still deep-link to the play page
  rather than the chat.

## Implementation notes

- Tables: `chat_conversations`, `chat_dm_participants`, `chat_messages`, `chat_read_states`
  (migration `20260629000000_chat.sql`).
- Conversation list queries (`ListDMConversationsByUser`, `ListPlayConversationsByUser`) each embed
  the unread-count subquery, the block filter (DMs) / roster filter (plays), the last-message
  lateral join, and the composite cursor condition; the API merges, sorts, truncates, and emits
  `next_cursor` as an opaque `RFC3339,id` string.
- Messages endpoint: `GET /conversations/{id}/messages?limit&before_id`, `id DESC` then reversed;
  `before_id=0` means newest page.
- Read marking happens in the conversation page `load` after fetching, and the load zeroes the open
  conversation's `unread_count` in the returned list (fetched before marking).
- Web routes: `/chat` (list, empty right pane on wide), `/chat/[id]` (conversation; list hidden on
  narrow). A `+server.ts` proxy under `/chat/[id]/messages` and `/chat/conversations` exposes
  pagination to the browser, since the typed API client is server-only (`API_BASE_URL` private
  env).
- Chat routes flip the root layout to `h-[100dvh]` and omit the footer; the conversation column
  escapes the wrapper padding (`-my-4`) so messages run to the edges under `h-20` fade gradients.
- Frontend components live in `web/src/lib/components/chat/`: `chat-page` (shell),
  `conversation-list` / `conversation-header` / `conversation-avatar`, `message-list`,
  `message-bubble`, `message-composer`, `link-confirm-dialog`, with `message-format.ts` (parser
  registry), `pending-messages.svelte.ts` (optimistic store), `link-confirm.svelte.ts` (dialog
  state), and `refresh-submit.ts` (in-place refresh submit handler).
- Message bubbles: DMs show no sender names (alignment suffices); play chats label incoming bubbles
  only; timestamps float on the last text line, telegram-style. Delete is a dropdown on
  click/right-click of own bubbles, submitting a hidden form.
- Composer: Enter sends, Shift+Enter newlines, IME composition respected; empty submits are
  cancelled client-side.
- Entry points: `[username]` profile `?/message` action and play page `?/chat` action (redirects
  with `?from=play`, which hides the mobile back-to-list arrow); chat notifications deep-link to
  `/chat/{id}` and force `invalidateAll` when that conversation is already open.
- Play chat entry is shown to viewers matching the backend membership rule
  (`canManage || viewer_state in (creator, confirmed, added)`).
